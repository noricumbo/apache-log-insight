# Setup Guide

## Overview

This guide shows a simple way to deploy the dashboard on a generic Apache server with Python 3 and optional Fail2Ban.

The examples use placeholders only:

- `logs.example.com`
- `/var/www/log-dashboard/public`
- `/var/log/httpd/access_log`
- `YOUR.PUBLIC.IP.HERE`

## Requirements

- Apache or `httpd`
- Python 3
- Static web serving
- Optional Fail2Ban

## Parser setup

Run the parser against an Apache access log and write the dashboard summary into the public data directory.

Example:

```bash
python3 parser/parse_logs.py \
  --log /var/log/httpd/access_log \
  --out /var/www/log-dashboard/public/data/log-summary.json \
  --limit 5000
```

## Dashboard setup

Serve the `public/` directory as static content. The dashboard should load:

- `/data/log-summary.json` for live output
- `/data/log-summary.sample.json` as a fallback for demos and local testing

## Cron setup

To refresh the summary every minute, use a cron entry similar to:

```cron
* * * * * /usr/bin/python3 /path/to/repo/parser/parse_logs.py --log /var/log/httpd/access_log --out /var/www/log-dashboard/public/data/log-summary.json --limit 5000
```

Adjust paths for your environment.

## Apache vhost setup

Point the dashboard vhost to the public directory:

```apache
<VirtualHost *:443>
    ServerName logs.example.com
    DocumentRoot /var/www/log-dashboard/public
</VirtualHost>
```

Keep the dashboard read-only from the browser's perspective. The JSON file should be generated by the parser, not edited through the web server.

## Optional Basic Auth

For a private dashboard, Basic Auth can add a simple access layer. Use placeholder paths and credentials in public examples, and keep real auth files out of the repository.

## Optional IP allowlist

You can further restrict access to known source networks or a trusted public IP address such as `YOUR.PUBLIC.IP.HERE`.

Treat allowlists as optional defense in depth, not the only control.

## Default deny vhost for unknown hosts

A catch-all default deny vhost can reduce noise from random host header scans and unknown domains. Load order matters, so place the default deny vhost early in the Apache vhost sequence.

## Troubleshooting

- The dashboard is blank: verify the JSON file exists and is valid.
- The page only shows sample data: verify the live JSON path is correct and web-accessible.
- The parser writes nothing: confirm the log format matches what the parser expects.
- Fail2Ban is not matching: test the regex against the live log format with `fail2ban-regex`.
- Unknown hosts still resolve: verify Apache vhost load order and default vhost behavior.

## Public release checklist

Before publishing the repository:

1. Remove real logs and generated production summaries.
2. Replace live examples with sanitized sample data.
3. Confirm docs, configs, and sample JSON use placeholder domains and documentation IP ranges only.
4. Re-run privacy grep checks across the repository.
5. If sensitive history was committed previously, consider publishing from a clean git history.
